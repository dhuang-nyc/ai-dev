# Capy AI

An AI-powered development team that takes a project idea from concept to merged pull requests. A Tech Lead agent collaborates with you to define a tech spec and task backlog, then autonomous Dev Agents pick up tasks, write code, and open pull requests — with full GitHub integration throughout.

**Live demo:** [capyai.daisyhuang.dev](https://capyai.daisyhuang.dev)

---

## How It Works

1. **Chat with the PM** — describe a rough idea to the Product Manager agent. It guides you through four structured stages (problem discovery → solution design → PMF check → brief), asking one focused question at a time.
2. **Finalise the brief** — the PM crystallises the scope into a structured product brief and hands it off to the Tech Lead once you confirm.
3. **Iterate on the spec** — the Tech Lead produces a structured TechSpec and a prioritised task backlog. You can refine either through conversation at any point.
4. **Approve & start** — approving the spec triggers GitHub repo creation and moves the project to `In Progress`.
5. **Run Dev Agents** — one agent per available workspace picks up an unblocked task, runs Claude Code inside a cloned repo, commits the implementation, and opens a pull request.
6. **Review & merge** — leave a comment on the PR to request changes or ask questions; the Dev Agent responds automatically. Merging closes the task and the Project Manager assigns the next one.

### Key Components

**Tech Lead Agent** (`src/team/agents/team_lead.py`)
Powered by `claude-opus-4-6` with extended thinking enabled. Runs in an agentic tool-use loop — it can read and modify tasks, update the tech spec, abort superseded work, and change project status. Conversation history is reconstructed from the database on every turn (stateless).

**Dev Agent** (`src/team/agents/dev_agent.py`)
Invokes `claude --print` (Claude Code CLI) inside a cloned GitHub repository. Claude Code is given a `SKILL.md` workflow guide and a detailed implementation prompt generated by the Tech Lead. It creates a branch, writes code, commits, pushes, and opens a PR — outputting `PR_URL: <url>` at the end so the task is linked.

**PM Agent** (`src/team/agents/product_manager.py`)
Powered by `claude-sonnet-4-6` with a tool-use loop. Walks the user through four structured stages — problem discovery, solution design, PMF assessment, and brief finalisation — asking one focused question at a time. Uses two tools: `finalize_brief` (captures the full product brief) and `start_project` (creates the `Project`, boots the Tech Lead conversation, and links the PM conversation to the new project). The PM agent is stateless; history is reconstructed from `PMConversation` / `PMMessage` on every turn.

**Project Manager** (`src/team/tasks.py`)
A lightweight Celery task that fires after every PR merge. It finds unblocked pending tasks and available workspaces, then enqueues the next round of Dev Agent runs automatically.

**PR Feedback Loop** (`/api/github/webhook/`)
GitHub webhook events for PR comments trigger `answer_pr_question`. The Dev Agent re-enters the workspace, reads the comment, and either implements the requested change (pushing to the same branch) or answers the question with a PR comment — using a keyword heuristic to distinguish between the two.

### Data Model

```
PMConversation ──── PMMessage[]
    │ (OneToOne, set after start_project)
    ▼
Project ──── Conversation ──── Message[]
    │
    ├──── TechSpec (versioned markdown)
    │
    └──── DevTask[] ──── blocked_by (self-referential M2M)
                │
                └──── Workspace (OneToOne, active tasks only)
```

Project lifecycle: `draft → planning → approved → in_progress → completed / aborted`

Task lifecycle: `pending → in_progress → pr_open → done / aborted`

## Getting Started

**Prerequisites:** Docker + Docker Compose, an Anthropic API key.

```bash
git clone <repo>
cp .env.example .env          # fill in ANTHROPIC_API_KEY at minimum
docker compose up --build
```

Open [http://localhost:5173](http://localhost:5173) — login with `admin` / `admin`.

For GitHub integration (automated repo creation, PRs, webhook), fill in the `GITHUB_*` and `DEV_GITHUB_*` variables in `.env`. See `.env.example` for details.

### Production deployment

The repo includes `docker-compose.prod.yml` targeting `capyai.daisyhuang.dev` with Traefik labels for automatic TLS. Copy `.env.example` to `.env.prod`, fill in all variables, then:

```bash
docker compose -f docker-compose.prod.yml up -d --build
```

## Future Work

### Reliability & observability
- **Real-time agent logs** — stream Dev Agent output to the database line by line rather than buffering the full run; surface a live log view in the UI per task.
- **Retry & error recovery** — exponential backoff for failed Celery tasks; dead-letter queue for tasks that exhaust their retry budget with an alert in the dashboard.
- **Metrics** — expose Celery queue depth, active worker count, and agent error rates as a lightweight stats panel.
- **Cost tracking** — record input/output token counts from every API call and store them against the task and project that triggered them. Surface accurate, granular cost estimates in the dashboard (e.g. "vocabulary-builder · $4.82 total — $3.10 Tech Lead, $1.72 Dev Agents across 6 tasks") so costs are visible, attributable, and actionable rather than only discoverable after the fact in the Anthropic developer console.

### Developer experience
- **Streaming chat** — replace the 2-second polling loop with server-sent events to stream Tech Lead responses token by token.
- **Spec diffs** — show a rendered diff between TechSpec versions so the user can see exactly what the Tech Lead changed after a scope update.
- **Branch previews** — embed a preview environment link per PR directly in the task card when the target repo has preview deploys configured.

### AI quality
- **PM ↔ Tech Lead continuity** — surface the PM's product brief inside the Tech Lead chat so the engineer can reference the original problem statement, target user, and PMF rationale while iterating on the spec.
- **Cache & token optimisation** — apply prompt caching to the Tech Lead's system prompt and conversation history (which can grow very large) to cut latency and cost on repeated turns; audit token usage across both agents and trim redundant context, verbose tool descriptions, and oversized history windows.
- **Feedback loop** — add thumbs up/down on Dev Agent PRs and use the signal to iteratively improve `SKILL.md` prompts.
- **Multi-model routing** — use a faster model (e.g. Haiku) for routine tool calls (list tasks, status checks) and reserve Opus for planning and spec generation.
- **Structured evaluation** — automated evals that run a known project idea end-to-end and score the output spec and generated code against a rubric.

### New agents
- **Deployment Agent** — after a project is marked complete, a dedicated agent provisions infrastructure, writes CI/CD pipelines, and deploys the application to a target environment (e.g. the same Oracle Cloud instance via Docker Compose + Traefik), posting the live URL back to the project.
- **UX Design Agent** — before the Dev Agent writes code, a design agent generates wireframes or component specs (using a tool like Claude's vision capabilities or a design API), producing a visual reference attached to each frontend task that Dev Agents and human reviewers can reference.

### Product
- **Multi-user support** — per-user project isolation; invite collaborators to a project with role-based access.
- **Notifications** — Slack or email alerts when a PR is opened, needs review, or a Dev Agent errors out.
- **CI integration** — have Dev Agents react to failing CI runs, not just PR comments, closing the feedback loop fully within GitHub Actions.
- **Dynamic workspace pools** — replace the fixed workspace list with ephemeral containers spun up per task and torn down after the PR merges, removing the bottleneck of a fixed agent count.
